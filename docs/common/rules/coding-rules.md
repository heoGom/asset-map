# Asset Map Coding Rules

Asset Map 프로젝트의 백엔드 및 프론트엔드 개발 시 준수해야 할 코딩 규칙입니다. 코드의 가독성, 유지보수성, 그리고 기술적 일관성을 유지하는 것을 목적으로 합니다.

---

## ☕ 백엔드 개발 규칙 (Backend Rules)

### 1. 아키텍처 및 설계
- **기능 기반 패키지 (Feature-based Packaging)**: 기술 계층(Controller, Service)이 아닌 비즈니스 기능(Account, Holding) 단위로 패키지를 구성합니다.
- **Thin Controller, Rich Service**: 컨트롤러는 요청 수신과 응답 반환에 집중하고, 복잡한 비즈니스 로직은 서비스 계층에서 처리합니다.
- **엔티티 노출 금지**: API 응답으로 JPA 엔티티를 직접 반환하지 않습니다. 반드시 DTO를 통해 필요한 데이터만 전달합니다.

### 2. 코드 스타일
- **Lombok 활용**: `@Getter`, `@RequiredArgsConstructor` 등을 사용하여 보일러플레이트 코드를 최소화합니다.
- **명시적 선언**: 자동 구성(Auto-configuration)에 의존하기보다 필요한 설정은 명시적으로 빈(Bean)으로 등록하는 것을 권장합니다.
- **상태 관리**: 도메인의 상태 변경은 엔티티 내부의 의미 있는 메서드(예: `updatePrice()`)를 통해 수행합니다.

---

## 🎨 프론트엔드 개발 규칙 (Frontend Rules)

### 1. UI 및 UX 디자인
- **반응형 최우선 (Responsive First)**: 모든 UI는 360px 너비부터 데스크톱 해상도까지 자연스럽게 대응해야 합니다.
- **모바일 우선 접근**: Tailwind CSS의 중단점(`sm:`, `md:`, `lg:`)을 활용하여 모바일 뷰를 먼저 설계합니다.
- **접근성**: 버튼과 링크의 터치 영역을 충분히 확보하고, 다크/라이트 모드 모두에서 가독성 높은 대비를 유지합니다.

### 2. 컴포넌트 개발
- **관심사 분리**: 데이터 페칭 로직(Custom Hooks)과 UI 렌더링 컴포넌트를 분리합니다.
- **Tailwind 활용**: 인라인 스타일 대신 Tailwind CSS 클래스를 사용하며, 반복되는 디자인은 공통 컴포넌트화합니다.
- **Layout Integrity**: 콘텐츠 넘침으로 인한 가로 스크롤 발생을 엄격히 금지합니다.

---

## 📝 공통 준수 사항
- **네이밍 컨벤션**: 
  - 클래스/컴포넌트: `PascalCase`
  - 변수/메서드: `camelCase`
  - 파일명: `kebab-case.md` (문서), `PascalCase.tsx` (컴포넌트)
- **주석**: 복잡한 비즈니스 로직이나 알고리즘에는 의도를 설명하는 주석을 남깁니다.
- **중복 배제 (DRY)**: 공통으로 사용되는 로직이나 타입은 `common` 패키지 또는 `lib/` 디렉토리에 정의하여 재사용합니다.
