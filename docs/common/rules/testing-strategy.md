# Asset Map Testing Strategy

Asset Map 프로젝트의 품질을 보장하고 안정적인 기능을 제공하기 위한 테스트 전략입니다. 모든 코드 변경은 검증된 테스트를 통해 그 정당성을 확보해야 합니다.

---

## 🏗️ 테스트 계층 및 범위

### 1. 단위 테스트 (Unit Test)
- **도메인 모델**: 엔티티 내의 비즈니스 메서드 및 도메인 제약 조건을 검증합니다.
- **유틸리티**: `MoneyCalculator`와 같은 계산 로직의 정확성을 `BigDecimal` 케이스별로 검증합니다.

### 2. 통합 테스트 (Integration Test)
- **서비스 계층**: `@SpringBootTest`를 활용하여 여러 컴포넌트 간의 상호작용과 데이터베이스 영속성을 검증합니다.
- **API 계층 (MockMvc)**: 컨트롤러의 응답 규격, HTTP 상태 코드, 입력값 검증(`@Valid`)을 테스트합니다.

---

## 🛠️ 핵심 규칙 (Core Rules)

- **테스트 필수 원칙**: 모든 신규 도메인과 비즈니스 로직은 최소 하나 이상의 테스트 케이스를 포함해야 합니다.
- **응답 구조 검증**: API 테스트 시 반환값이 `ApiResponse<T>` 규격을 준수하는지, 에러 시 정확한 `ErrorCode`가 포함되는지 확인합니다.
- **격리된 환경**: 테스트는 로컬 H2 데이터베이스를 사용하며, 각 테스트 메서드는 독립적으로 실행되어야 합니다.
- **검증 케이스**:
  - **성공(Positive)**: 정상적인 입력에 대한 올바른 결과 반환.
  - **예외(Negative)**: 잘못된 입력(Bad Request), 존재하지 않는 자원(Not Found) 등에 대한 명확한 에러 응답.
  - **계산(Calculation)**: 자산 합계 및 비중 계산 시 소수점 처리 로직 검증.

---

## 📡 API 테스트 체크리스트

| 항목 | 검증 내용 | 기대 결과 |
| --- | --- | --- |
| **Response Wrapper** | 응답이 `status`, `message`, `data` 필드를 포함하는가? | `SUCCESS` 또는 `ERROR` 상태 확인 |
| **Validation** | 필수 필드 누락 시 적절한 응답을 주는가? | `400 Bad Request` |
| **Business Logic** | 도메인 제약 조건(예: 잔액 부족 등)을 위반했는가? | `BusinessException` 및 커스텀 에러 코드 |
| **Data Integrity** | DB 저장 후 조회 시 데이터가 일치하는가? | 저장된 필드 값 비교 검증 |

---

## 🚀 개발 워크플로우 (TDD-ish)

1.  **요구사항 분석**: 구현할 기능과 예외 상황을 정의합니다.
2.  **테스트 시나리오 작성**: `docs/project/api-test-scenarios.md`에 수동 테스트 케이스를 먼저 정리할 수 있습니다.
3.  **구현 및 테스트**: 코드를 작성하고 단위/통합 테스트를 통해 검증합니다.
4.  **검증**: `./gradlew test`를 실행하여 전체 테스트 통과 여부를 확인합니다.
